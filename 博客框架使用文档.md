## 1. 项目概览

本项目基于 Astro 构建，内容采用 Content Collections（`src/content/blog`）统一管理；站点信息、导航、标签色板等集中在配置文件中维护。

## 2. 目录结构

```text
/
├── public/                         # 静态资源（不经构建直接发布）
│   ├── favicon.svg
│   ├── fonts/
│   └── icons/                      # SVG/PNG 图标
├── src/
│   ├── config/
│   │   └── site.ts                 # 站点信息（标题/作者/导航/标签色板/项目）
│   ├── content/
│   │   └── blog/                   # 文章（.md/.mdx）
│   ├── components/                 # 组件（页头/页脚/邮件复制等）
│   ├── layouts/                    # 页面布局（全站壳、文章页布局等）
│   ├── pages/                      # 路由页面（index/blog/archive/tags/rss/sitemap）
│   ├── styles/                     # 样式
│   └── data/                       # 数据（经历等）
├── .github/workflows/deploy.yml     # GitHub Pages 部署
├── package.json
└── astro.config.mjs
```

## 4. 文章管理

### 4.1 新增文章

1. 在 `src/content/blog/` 下新建 Markdown 文件，例如：
   - `src/content/blog/your-post-slug.md`
2. 按下述 Front-matter 模板填写必要字段。

### 4.2 Front-matter 必填/推荐字段

项目的文章 schema 以 [content.config.ts](file:///c:/Users/Administrator/OneDrive/Projects/widechaos-blog/src/content.config.ts) 为准，建议使用以下模板：

```yaml
---
title: "文章标题"
description: "文章摘要（用于列表与 SEO）"
pubDate: "2026-03-18"
author: "邬仁超"
tags: ["RAG", "Agent", "云原生"]
draft: false
readTime: "8 分钟"
updatedDate: "2026-03-18"
---
```

- `draft: true` 会让文章不进入 RSS、Sitemap 与列表。
- `updatedDate` 用于 sitemap 的 `<lastmod>`（如不填默认使用 `pubDate`）。

### 4.3 文章文件名与 URL 规则

- 文件名即 slug：`src/content/blog/xxx.md` → `/blog/xxx/`
- 建议用英文/数字/短横线作为 slug，稳定、易迁移。

### 4.4 图片与静态资源放置规则

优先将图片放在 `public/` 下，按文章时间或主题分目录，例如：

- `public/images/2026/03/your-post/cover.png`
- 文中引用：`/images/2026/03/your-post/cover.png`

这样图片无需被打包处理，构建与部署更简单，路径也更稳定。

### 4.5 更新/下线文章

- 更新正文：直接改 `.md` 内容，同时建议更新 `updatedDate`
- 下线文章：把 `draft` 置为 `true`（页面不会出现在列表/RSS/Sitemap）

## 5. 版本管理规范（Git）

### 5.1 分支策略（推荐）

- `main`：线上稳定分支
- `dev`：日常集成分支
- `feat/<topic>`：功能/文章分支（例如 `feat/rag-observability`）
- `fix/<topic>`：修复分支

### 5.2 提交信息模板（推荐）

```text
<type>(<scope>): <subject>

<body>
```

- `type`：feat / fix / chore / docs / refactor
- `scope`：blog / seo / ui / build

### 5.3 回滚步骤（示例）

1. 定位回滚目标提交：`git log --oneline`
2. 生成回滚提交：`git revert <commit_sha>`
3. 推送并触发部署：`git push origin main`

如需“强回退到某个历史点”，使用 `git reset --hard` 前务必确认团队协作方式与远端策略。

## 6. 自动化部署

### 6.1 GitHub Pages（项目已配置）

- 工作流文件：[deploy.yml](file:///c:/Users/Administrator/OneDrive/Projects/widechaos-blog/.github/workflows/deploy.yml)
- 触发条件：推送到 `main` 分支会自动构建并发布 `dist/`

### 6.2 Vercel（示例工作流）

可使用 GitHub Actions 调用 Vercel CLI 部署，常用 Secrets 命名如下：

- `VERCEL_TOKEN`
- `VERCEL_ORG_ID`
- `VERCEL_PROJECT_ID`

示例（仅供参考）：

```yaml
name: Deploy to Vercel
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - run: npm run build
      - run: npx vercel --prod --token ${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
```
